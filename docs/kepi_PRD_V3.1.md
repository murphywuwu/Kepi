## 1. 项目定位

**项目定位**：AI 驱动的“微肉鸽（Roguelite）+ 情感自走棋”。

**核心体验**：在结合 AI 编排的归乡路线中，通过极限自走棋战斗与残酷典当抉择，体验客家先民的羁绊与乡愁。

**单局时长**：约 10–12 分钟。

**核心支柱**：

- 路线层：固定结构的归乡航线图承载情绪节奏。
- 战斗层：死保水客的极限自走棋负责强机制检验。
- 抉择层：客批与未来通关阈值都可以被典当，形成高风险战略杠杆。
- AI 层：AI 只填充叙事文本，不参与主流程、状态判定或节点跳转。

---

## 2. 核心玩法大盘（Gameplay Loop）

V3.1 全面采用“Roguelike 线性卷轴图 + 核心战斗检验”的结构。

```txt
开局
  │
  ▼
归乡路线节点 1
  │
  ├─ 战斗节点 → 自走棋战斗 → 结算
  ├─ 客批典当行 → 双向典当 → 资源/阈值变化
  └─ 篝火夜话 → AI 文本二选一 → 本地效果结算
  │
  ▼
重复 6–8 个节点
  │
  ▼
终点胜负判定 → 风浪抓信结局演出
```

### 2.1 局外核心：伪随机归乡路线（The Journey）

**呈现形式**：一张从“南洋”指向“故乡”的单向航线图，使用极简卷轴 UI。

**路线长度**：6–8 个节点。

**工程妥协**：路线结构写死在本地，保证黑客松现场情绪曲线不崩盘。AI 不负责逻辑节点生成，只负责内容填充。

**推荐结构样例**：

```txt
篝火夜话 → 战斗 → 客批典当行 → 战斗 → 篝火夜话 → 极难战斗 → 终点
```

### 2.2 节点类型

| 节点 | 玩法 | 本地职责 | AI 职责 |
|---|---|---|---|
| 战斗节点 | 进入核心自走棋局内 | 生成敌人、开战、结算胜负与资源 | 不参与 |
| 客批典当行 | 花钱买命，或卖命换钱 | 执行典当规则、资源变化、阈值变化 | 不参与 |
| 篝火夜话 | 纯文本二选一抉择 | 维护选项效果白名单并结算 | 仅生成文案 |

## 3. 局内核心：极限自走棋（The Battle）

### 3.1 死保水客

水客是产出“客批”与“桑梓值”的唯一核心单位：

- 水客无战斗力。
- 水客必须在战场上，可被攻击和击杀。
- 玩家需要用肉身筑墙保护水客。
- 战斗胜利且水客存活，才产出客批与桑梓值。

### 3.2 手势盲盒（首尾呼应）

每场战斗开局，天上随机飘落“乡音符”（开局 Buff）。玩家需要通过摄像头伸手“抓取”来激活。

**降级方案**：

- 无摄像头或权限失败时，使用鼠标 / 触控点击抓取。
- 路演设备不稳定时，默认启用 pointer 模式，保留手势开关。

### 3.3 Juice 大招：落叶归根

凑齐【宗族】最高级羁绊时，触发全屏视觉高潮“落叶归根”：

- 全体获得高额攻速。
- 全体获得吸血。
- 屏幕出现落叶、归潮、土楼灯火呼应的全屏高光。

该机制是宗族羁绊的终局爽点，不应替代基本数值平衡。

## 4. 三大核心系统融合设计（Ely 的黄金法则）

### 4.1 经济系统：极具策略深度的双向典当

V3.1 最终抛弃冗余利息系统，将“典当”打造为最高级的高风险战略杠杆。

**基础收入**：每过一个节点获得固定微薄底薪。想爆发必须典当。

**典当仪式感**：

- 点击典当时，全屏暗下。
- 一封写有“家常”的客批出现在屏幕中央。
- 玩家需长按鼠标或手势，亲眼看着信件在火焰中一点点烧成灰烬。
- 随后金币掉落，完成资源兑现。

#### 4.1.1 策略分支 A：防守反击

| 项 | 规则 |
|---|---|
| 条件 | 手里必须 ≥ 1 封已获得的客批 |
| 代价 | 扣除 1 封客批，通关进度倒退 |
| 收益 | 立即获得 15 金币 |
| 定位 | 残局苟活、临时补强、防守反击 |

#### 4.1.2 策略分支 B：极致进攻

| 项 | 规则 |
|---|---|
| 条件 | 任何时候均可使用，哪怕手里 0 封信 |
| 代价 | 通关所需客批阈值永久 +1 |
| 收益 | 立即获得 35 金币 |
| 定位 | 卖未来换当下，开局组出强阵容打连胜 |

**设计意图**：这是“卖血理财”。玩家赢在当下，但未来的路更长、更险。

#### 4.1.3 关键状态字段建议

```ts
type PawnState = {
  pawnedKebi: number;          // 绝境当信累计次数
  bloodDebtCount: number;      // 泣血透支累计次数
  kebiThreshold: number;       // 基础 5 + bloodDebtCount
};
```

### 4.2 养成系统：土楼的反哺

**桑梓值流转**：

```txt
自走棋战斗胜利
  且水客存活
    → 产出桑梓值
    → 自动注入地图背后的土楼修复进度条
    → 达成阈值后反哺后续战斗
```

| 修复度 | 阶段 | UI 表现 | Gameplay 反哺 |
|---|---|---|---|
| 33% | 修缮 | 水井出水 | 全军开局获得最大生命值 20% 的阵型护盾 |
| 66% | 翻新 | 高墙立起 | 全军获得 15% 攻速加成 |
| 99% | 焕然 | 灯火通明 | 解锁“不屈”被动：全体一次致死免疫并无敌 1.5 秒 |

### 4.3 AI 叙事系统：机制旁白与夜话文本

AI 只负责叙事内容填充，不负责规则判断。

**输入只允许本地标签化传参**：

```ts
type NarrativeTags = {
  deathCount: number;
  didPawn: boolean;
  didBloodDebt: boolean;
  winStreak: number;
  waterGuestSurvived: boolean;
  homeRepairMilestone?: 33 | 66 | 99;
};
```

**输出只允许纯文本字符串**：

```ts
type NarrativeOutput = {
  text: string;
};
```

## 5. AI 工程与边界熔断机制

为了保障黑客松现场 5 分钟体验的绝对流畅，大模型 API 调用必须遵守以下铁律。

### 5.1 核心红线

- 游戏主流程 100% 由本地代码掌控。
- 状态判定 100% 由本地代码掌控。
- 节点跳转 100% 由本地代码掌控。
- AI 不能生成、决定或修正任何关键状态。
- AI 不能输出需要客户端解释执行的规则 JSON。

### 5.2 标签化传参

游戏结算时，本地代码只向大模型传递预设标签，例如：

```txt
Death_Count: 3
Did_Pawn: True
Did_Blood_Debt: False
Win_Streak: 2
WaterGuest_Survived: True
```

AI 只负责根据标签输出一段机制叙事旁白。

### 5.3 1.5 秒熔断兜底（P0）

| 要求 | 说明 |
|---|---|
| 本地预设 | 策划提前写死 3 组本地文案：惨胜、苟活、碾压 |
| 超时限制 | 客户端调用 AI 接口最高 1500ms 超时 |
| 报错策略 | 一旦超时或接口报错，不阻断画面 |
| UI 策略 | 无缝顶上本地预设文案 |
| 目标 | 评委看到的是连续叙事，而不是等待和报错 |

## 6. 结局演出的收束（The Climax）

### 6.1 胜利判定

走到归乡路线终点后统一判定：

```txt
胜利 = 存续度 > 0 且 手中客批 >= 最终阈值
最终阈值 = 基础 5 封 + 泣血透支惩罚附加数
```

### 6.2 演出表现

- 触发“风浪”动画。
- AI 生成的信件与真实馆藏客批漫天飞舞。
- 子弹时间开启。
- 玩家通过摄像头在风浪中拼命抓取。
- 无摄像头时降级为 pointer 抓取。

### 6.3 终极情感字幕

> 在这场漫长的对抗中，你没能赢下所有的期冀，但你让一个客家人的牵挂，回了家。

---

## 7. V3.2 / V3.3 优化方向

后续版本优先优化三件事：

1. 增加少量战斗规则变化，提升每场战斗差异性。
2. 让 AI 夜话的选择直接影响下一场战斗或资源。
3. 强化最后的玩家交互收束，例如亲手送出最后一封客批，让情感高潮由玩家完成，而不是主要依赖旁白。

这些方向应在不明显增加开发成本的前提下推进，避免重新扩大系统复杂度。

## 8. 与 V2.0 的关键差异

| 维度 | V2.0 | V3.1 |
|---|---|---|
| 局外结构 | 6 关固定战斗流程 | 6–8 节点线性归乡路线 |
| 节点类型 | 基本只有备战 / 战斗 / 结算 | 战斗、客批典当行、篝火夜话 |
| 经济核心 | 典当客批换 15 金币 | 双向典当：当信 + 透支未来 |
| AI 落点 | 数字客批 / 结局文本 | 机制旁白 + 夜话文本，严格熔断 |
| 手势交互 | 主要用于结局抓信 | 开局抓乡音符 + 结局抓信首尾呼应 |
| 羁绊高光 | 宗族攻击加成 | 宗族最高级触发“落叶归根”Juice 大招 |
| 胜利阈值 | 固定 5 封 | 基础 5 封 + 透支惩罚 |

## 9. 当前 Ely 分支落地风险分析

本节基于 2026-07-01 Ely 分支当前代码与文档状态，仅记录会影响 V3.1 落地的风险。

### 9.1 P0 风险：路线节点系统尚未存在

当前主流程仍是 `prep -> battle -> settlement -> ending` 的固定状态机，`src/data/stages.ts` 仍按 6 个战斗关卡生成，未抽象出 journey node、典当行节点或篝火夜话节点。

**影响**：

- V3.1 的“线性卷轴图”无法直接接入现有状态机。
- 典当行与夜话若硬塞进 `prep`，会导致状态边界混乱。
- AI 夜话选项缺少本地白名单效果表。

**建议**：

- 新增 `JourneyNode` 静态配置，先固定 6–8 节点。
- 在引擎层新增 `journeyIndex` 与 `currentNode`，不要让 UI 自己判断节点推进。
- 把战斗节点继续复用现有 battle 模块，典当行 / 夜话作为独立 phase 或 subphase。

### 9.2 P0 风险：双向典当尚未落地，且旧经济仍在

当前 `GameAction` 没有 `PAWN_KEBI` 或 `BLOOD_DEBT`，`GameState` 没有 `pawnedKebi`、`bloodDebtCount`。同时 `src/engine/economy/index.ts` 仍计算利息和连胜 / 连败奖励，`src/data/balance.ts` 中 `interestPerTenGold`、`maxInterest`、`streakBonuses` 仍有效。

**影响**：

- V3.1 的核心经济杠杆无法演示。
- 利息 / 连胜奖励会稀释“固定微薄底薪 + 典当爆发”的设计。
- 透支未来需要动态提高 `kebiThreshold`，当前胜利判定仍假设阈值固定。

**建议**：

- P0 删除或禁用利息、连胜 / 连败经济收益。
- 新增两条 engine action：`PAWN_KEBI`、`BORROW_AGAINST_RETURN`。
- 将 `kebiThreshold` 改为 `baseKebiThreshold + bloodDebtCount` 的可审计状态。

### 9.3 P0 风险：水客仍是后勤位，不参与战斗

当前 `SupportUnit` 仍是固定后勤槽位，`BattleInput` 只接收 `Piece[]`，战斗模拟中没有水客单位。`settleStage` 在胜利时无条件发放 `kebi +1` 与 `sangzi`，没有判断水客是否存活。

**影响**：

- “死保水客”核心体验尚不存在。
- “胜利但水客死亡不得客批 / 桑梓”的情感惩罚无法成立。
- 敌方刺客优先锁水客也缺少目标。

**建议**：

- 将水客、乡贤转为可上场单位，或新增 `BattleSupportUnit` 并进入 battle runtime。
- `BattleResult` 增加 `waterGuestSurvived`。
- `settleStage` 只在 `won && waterGuestSurvived` 时发放客批和桑梓。

### 9.4 P0 风险：AI 熔断时间与职责边界不符合 V3.1

当前客户端 AI 请求超时为 15 秒，服务端 provider 超时为 12 秒，远高于 V3.1 的 1500ms 红线。当前 AI 类型仍围绕 `digital-letter`，不是机制旁白或夜话文本。

**影响**：

- 路演现场弱网或 provider 慢响应会造成可见等待。
- AI 输出和 fallback 文案池不按“惨胜、苟活、碾压”等机制标签组织。
- 篝火夜话如果临时接 AI，容易越过“AI 只生成文本”的红线。

**建议**：

- 客户端请求改为 1500ms 熔断，服务端也应设置接近的 provider timeout。
- 新增 `narrative-caption` 与 `campfire-choice-copy` 两类请求，仍返回纯文本。
- fallback 文案池按标签组合分组，至少覆盖惨胜、苟活、碾压、典当、透支、水客死亡。

### 9.5 P0 风险：土楼 Buff 目前偏视觉，未进入战斗公式

当前 `homeRepair` 已能驱动视觉阶段和结算过场，但 battle 模块没有应用 33% 护盾、66% 攻速、99% 免死。现有土楼视觉还按 6 档 `stage1–stage6` 映射，与 V3.1 的 33 / 66 / 99 三个强机制阈值不完全一致。

**影响**：

- “家园反哺战斗”的核心反馈不成立。
- 玩家无法感知桑梓值的策略价值。
- 视觉阶段与机制阈值可能出现口径不一致。

**建议**：

- 在 battle 初始化时根据 `homeRepair` 注入 `homeBuffs`。
- 护盾、攻速、免死都应由 battle runtime 结算，UI 只展示。
- 保留 6 档视觉细化可以，但机制阈值必须明确映射到 33 / 66 / 99。

### 9.6 P1 风险：手势盲盒缺少局内入口

当前结局层已有 `GestureLayer`，支持 pointer / gesture 模式，但战斗开局没有“乡音符”掉落、抓取窗口、Buff 激活和超时降级。

**影响**：

- 首尾呼应只剩结尾，开局手势盲盒无法形成记忆点。
- 摄像头权限、设备兼容、路演现场光照会成为临场风险。

**建议**：

- P0 先做 pointer 抓取版本，手势识别作为增强。
- Buff 种类控制在 3 个以内，且全部本地枚举。
- 摄像头失败不弹阻断式错误，只显示可点击抓取。

### 9.7 P1 风险：宗族 Juice 大招缺少技能与表现层

当前数据中有 `clanSynergy` 攻击加成，但 battle 模块没有最高级羁绊触发大招、攻速 / 吸血临时状态，也没有“落叶归根”的全屏 FX。

**影响**：

- 宗族最高级缺少高光，战斗高潮不足。
- 如果直接叠加到现有攻击加成，可能造成数值不可控。

**建议**：

- 先定义清楚触发条件：唯一棋子类型数、上场人数、是否含水客。
- 大招效果建议有持续时间，避免永久滚雪球。
- 先实现数值，再补全屏表现。

### 9.8 P1 风险：文档口径存在旧版残留

当前 `docs/kepi_data-structures_v1.md` 仍基于 V1.6，`docs/kepi_assets-and-media-plan_v2.md` 内部又出现 4 回合 / 阈值 4 的 V2.0 素材口径，与 V3.1 的 6–8 节点、基础 5 封阈值不一致。

**影响**：

- 开发、素材与测试会按不同阈值推进。
- QA 难以判断“当前行为是 bug 还是旧设计”。

**建议**：

- V3.1 PRD 合入后，补一版 `kepi_data-structures_v2.md`。
- 素材计划新增 V3.1 小节，不直接覆盖历史版本。
- TODO 文档按 Feature 拆成 V3.1 迁移清单。

## 10. V3.1 P0 落地顺序建议

| 顺序 | 工作项 | 目标 |
|---:|---|---|
| 1 | 固化 JourneyNode 数据结构与本地路线 | 先让卷轴图能推进 |
| 2 | 禁用旧经济收益，加入双向典当 action | 建立核心抉择 |
| 3 | 水客进入战斗并改造结算 | 建立死保核心 |
| 4 | 土楼 Buff 进入 battle 初始化 | 让养成反哺可感知 |
