## 0. 执行口径

### 0.1 版本目标

- V2.0 不再是 V1.6 的长线自走棋压缩版，而是 4 回合、约 10 分钟的微型极限生存局。
- P0 目标：跑通一局完整 V2.0 主流程，玩家能明确体验到“护信 vs 保命”的核心抉择。
- P1 目标：把典当痛感、土楼反哺、水客危机做出足够强的路演表现。
- P2 目标：数值验证、AI 对手难度、更多视觉/音频精修。

### 0.2 开发顺序约定

每个 feature 内部按以下顺序推进：

1. 类型与数据
2. 引擎规则
3. Store 镜像与 action 分发
4. UI / Canvas / 音效接入
5. 单元测试与 E2E 验证

### 0.3 状态标记

- `[x]`：已由 V1 基础或本轮改造完成，且可被 V2 继续使用
- `[ ]`：待实现 / 待迁移 / 待验证
- `P0`：路演必须可用
- `P1`：强烈建议完成，提升表现力
- `P2`：工期允许时做

## Feature B - 极简经济与典当客批

> 删除安全理财，把玩家推到“烧信换钱救场”的抉择里。

### 目标验收

- 每回合固定工资 5 金币。
- 商店刷新 1 金币，升人口 4 金币。
- 玩家可在备战阶段典当 1 封客批，立即获得 15 金币，累计 `pawnedKebi +1`。
- 客批不足时不可典当；典当不可逆，可能导致最终客批不足 4。

- [x] P0 新增 `PAWN_KEBI` action
- [x] P0 实现典当 reducer：仅备战阶段、`kebi >= 1` 时可触发
- [x] P0 实现典当后 `kebi -1`、`gold +15`、`pawnedKebi +1`
- [x] P0 Store 暴露 `pawnKebi()` action

## Feature D - 死保水客与后勤上场

> 水客是客批唯一来源；玩家赢了但水客死了，也拿不到信。

### 目标验收

- 水客必须上场并存活，胜利后才获得 `+1 客批 +20 桑梓值`。
- 我方胜利但水客战死时，不扣存续度，但不获得客批与桑梓值。
- 我方失败时，无论水客状态如何，存续度 `-1`。
- 乡贤在场时，桑梓到家园修复的转化率提高 50%。

### 任务清单

- [x] P0 战斗快照记录水客是否参战、是否存活、是否战死
- [x] P0 结算逻辑改为“胜利 + 水客存活”才产出客批和桑梓
- [x] P0 实现“胜利但水客死亡”的特殊结算文案与收益为空
- [x] P0 实现失败扣存续度逻辑
- [x] P0 实现乡贤在场时桑梓转修复 `+50%`
- [x] P0 在结算事件中输出 `waterGuestDied` / `waterGuestSurvived`
- [x] P1 水客低血量视觉反馈：黄边、红边、脉冲暗角
- [x] P1 水客低血量听觉反馈：呼吸、心跳、BGM 压低

## Feature G - 结算、胜负判定与三结局

> 结局由存续度、客批数量与 4 回合完成情况共同决定。

### 目标验收

- 中途 `survival <= 0` 立即进入“风浪抢救”失败结局。
- 第 4 关结束后，`survival > 0 && kebi >= 4` 进入“完美归乡”。
- 第 4 关结束后，`survival > 0 && kebi < 4` 进入“遗憾留守”。
- 三种结局共用手势接信过场，但信封数量、信件内容、背景氛围不同。

### 任务清单

- [x] P0 结局叙事读取 `kebi`、`pawnedKebi`、`homeRepairTier`、水客结果
- [x] P0 更新结局信件数据，支持三种情绪调性
- [x] P0 接入三结局共享手势接信过场
- [x] P1 不同结局显示不同信封数量、背景氛围、字幕文案

## Feature H - AI 机制叙事旁白

> AI 不再承担“大内容生成”，而是在每回合结算后生成一句侨批风格旁白。

### 目标验收

- 每次战斗结算后收集关键事件，生成一句 10-25 汉字左右的侨批语气旁白。
- AI 不可用时使用本地 fallback 文案池。
- 相同事件组合可缓存复用。
- AI 密钥只在服务端 API 使用，不暴露给客户端。
